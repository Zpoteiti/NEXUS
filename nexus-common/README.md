# nexus-common

## 1. 一句话定位
nexus-common 是 NEXUS 的协议与常量基座，负责定义 Server 与 Client 的“共享语言”，并且该协议层已冻结。

## 2. 职责边界
### 负责什么
- 定义跨端消息协议结构（WebSocket 双向消息、工具请求与回传结构）。
- 定义跨端一致常量（协议版本、心跳间隔、工具输出截断阈值、退出码语义）。
- 定义跨端错误载荷结构与错误码枚举。

### 不负责什么
- 不负责任何业务流程编排（例如会话调度、鉴权流程、工具执行）。
- 不负责网络收发、数据库读写、模型调用。
- 不负责模块内部状态管理。

## 3. 架构决策（What + Why）
### 决策 A：协议冻结，向后兼容优先
- What：`nexus-common` 的消息结构、常量语义、错误码集合视为稳定契约，不做随意改动。
- Why：Server、Client、WebUI 由不同进程协同运行，协议漂移会直接造成跨进程不兼容；冻结契约可降低联调成本。

### 决策 B：统一使用 Device Token
- What：认证凭据格式固定为 `nexus_dev_` + 32 位随机字符。
- Why：单一机制最易审计与吊销；跨端实现简单，避免多套认证并存带来的状态不一致。

### 决策 C：工具参数与 Schema 统一为 `serde_json::Value`
- What：工具调用参数与工具声明采用动态 JSON 值传递，不引入额外强类型 Schema 结构体。
- Why：可兼容内置工具、MCP 工具、Skill 工具的异构输入，减少协议升级频率。

### 决策 D：`exit_code` 语义固定
- What：退出码固定为 `0 成功 / 1 失败 / -1 超时 / -2 取消 / -3 校验失败`。
- Why：使 Server、Client、LLM 对工具结果形成稳定共识，便于自动纠错与回放分析。

### 决策 E：工具输出双端截断策略固定
- What：最大输出 10000 字符，超限后保留前 5000 与后 5000，并在中间插入截断提示。
- Why：既保留上下文头尾关键线索，又控制传输体积，避免长输出拖垮链路与上下文窗口。

## 4. 与其他模块的关系
### 依赖谁
- 仅依赖 `serde` 与 `serde_json`，不依赖业务模块。

### 被谁依赖
- 被 `nexus-server`、`nexus-client` 直接依赖，作为跨端协议与常量来源。

### 通信方式
- 不直接通信；仅提供可序列化结构与常量定义，供上层通过 WebSocket/HTTP 使用。

## 5. 环境要求与运行方式
### 环境要求
- Rust 1.85+
- Rust edition 2024

### 运行方式
- 作为 workspace 内部库使用，无独立进程启动方式。
- 变更协议后必须同步更新 Server 与 Client 的对应处理逻辑并联调。
